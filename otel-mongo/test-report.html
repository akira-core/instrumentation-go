client_deliver_test.go 5 / 10 v1 ≡ v2 (different layout) (1) traced.ClientState.ShutdownDeliver with nil MongoTP (noop), real provider (invokes processor Shutdown), with bounded 3-second timeout when processor blocks; (2) TestClientDisabledPathHasNoDeliverState asserts mongoTracingEnabled() resolves false with explicit env=0; (3) v2 has additional initMongoProvider + StartDeliverSpan tests folded into the same file. cursor_test.go 7 / 10 v1 ≡ v2 Cursor.Decode direct/traced paths, SingleResult.Decode / Raw / TraceContext first-touch-ends-span contract with 6 cross-permutation sub-cases (decode×2, decode→raw, raw→decode, decode→tracectx, tracectx→raw, all-three), passthrough impl emits no spans. Construct cursors via mongo.NewCursorFromDocuments / mongo.NewSingleResultFromDocument — no live mongod needed. Build documents carrying _oteltrace with injectTraceIntoDocument helper. The "span ends exactly once on first of Decode/Raw/TraceContext" contract is easy to break silently — a future refactor that adds an early-return path could end the span twice (panic) or zero times (leak). The cross-permutation cases nail this down.
cursor_decodewithcontext_test.go (v1 only) 2 v1 only v1-specific behaviour: DecodeWithContext creates an INTERNAL span with a new TraceID linked to the origin trace (legacy semantic). Also asserts disabled path emits no span. This file is the only place where the legacy v1 contract is enforced; without it, v1 silently regresses to v2 behaviour without breaking any other test.
collection_test.go 14 v1 ≡ v2 Integration suite (testcontainers MongoDB): InsertOne/InsertMany/ReplaceOne/UpdateOne/DeleteOne/CountDocuments/UpdateByID/DeleteOneByID/FindOneByID/FindByIDs/BulkWrite, all asserting _oteltrace presence on writes + the global-off zero-spans path. Includes TestNewCollection for the strategy-split decision. tcmongo.Run(ctx, "mongo:7.0") starts a standalone MongoDB container in TestMain; all tests share it. requireMongoDB falls back to MONGO_URI env var or skips when Docker is unavailable. Without integration tests the wrapper has no end-to-end confidence — driver behaviour is too subtle (e.g., projection rules, atomic ops, write concerns) to be faithfully mocked. _oteltrace field presence on real reads is the only meaningful proof that the wrapper actually round-trips through MongoDB BSON serialisation.
database_test.go 3 v1 ≡ v2 Gate inheritance chain: ClientState.ForDatabase() → DatabaseState.NewCollection() propagates Tracer / Propagator / PropagationEnabled / DeliverTracer / ServerAddr / ServerPort verbatim. If the inheritance chain drops a field, a Database/Collection constructed via a Client could lose its propagation setting partway through, producing inconsistent _oteltrace behaviour between Client/Collection used directly vs via Database.Collection.
internal/shared/extract_metadata_test.go 7 + 1 bench v1 ≡ v2 ExtractMetadataFromRaw direct-lookup parity with old reflection path, missing field, wrong-type subdocument, empty traceparent; bson.D reverse-scan: position invariance (head/middle/tail), reverse-scan finds LAST occurrence (locks in the optimisation), benchmark comparing tail vs head lookup cost.
