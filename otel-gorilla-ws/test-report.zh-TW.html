Unit 測試（root package）

檔案 測試數 測什麼 怎麼測 為何測
env_flags_test.go 1 2-tier flag gate 真值表：OTEL_INSTRUMENTATION_GO_TRACING_ENABLED × OTEL_GORILLA_WS_TRACING_ENABLED。約 20 列涵蓋 truthy/falsy 同義詞。無 propagation flag（設計如此 — envelope 是 inline；subprotocol negotiation 替代該角色）。 直接呼叫 wsTracingEnabled()；每列 t.Setenv + os.Unsetenv；每列後 resetWSGateForTest()。 這是 newConn 中 per-connection impl 選擇所依賴的承重 gate。回歸會讓每個新建 Conn 走錯路徑（或走對但不該 tracing）。
message_test.go 9 跨語言 wire format 契約：MarshalWire 產生 JS 端所期待的精確 {"header":{...},"data":...} 形狀；TryUnmarshalWire 同時接受 envelope 與 legacy flat format（向後相容）；非 JSON payload 包成 JSON-encoded 字串；sync.Pool buffer 跨 goroutine 重用安全。 手工 byte 陣列用 assert.JSONEq 比對；goroutine fan-out 對 pool reuse 壓測。 這個檔是跨語言 wire 契約的唯一守護者。任何 envelope 形狀改動都會悄悄破壞 JS 客戶 — 不架設 JS test rig 的話，唯一抓得到的方法是 Go 端的精確 byte-level 斷言。
options_test.go 5 resolveOptions 在未傳 option 時 fallback 到全域 Tracer/Propagator；自訂 option 不污染全域；NewConn 在 feature disabled 時選 internal/direct.Conn，enabled 時選 internal/traced.Conn；negotiated=false（passthrough subprotocol）保留 tracing 但關閉 PropagationEnabled。 Recorder-based：斷言自訂 option tracer.Start 寫到哪個 TP vs 全域 TP。 Functional option 是呼叫者客製 wrapper 而不 monkeypatch 全域的唯一方法。若 resolveOptions 錯誤地 mutate 全域，用不同 propagator 同時 instrument 多條連線會有順序依賴行為。
helpers_test.go 4 錯誤型別判定（close error 偵測、normal vs abnormal closure）、format helper。 直接呼叫 helpers.go 中的 helper。 這些 helper 在錯誤路徑執行 — bug 在真實 disconnect 發生前看不見。直接 unit 覆蓋抓到 close-code 分類的 off-by-one。
conn_test.go 9 (1) TestRoundTrip + TestSpanAttributes — 端到端 envelope round-trip 與 span 屬性驗證；(2) Scenarios C、D、E（client-side dial 情境 — 見 tests/integration/handshake_test.go 的完整 A–H 表）；(3) Scenarios F、G、G_FromPrefixedClientToken、H（server-side upgrader 情境）。 httptest.NewServer 配合 otelgorillaws.Upgrader 與原生 websocket.Upgrader 兩種 handler；tracetest.SpanRecorder 觀察 span。 Subprotocol negotiation 是這個 wrapper 最核心的正確性議題 — 共有 8 種獨立情境（A–H per README）。每個都是 upgrade path 中的一個決策。沒有 per-scenario 測試的話，某個分支的回歸要等到特定客戶組合在 production 命中才會浮現。
upgrader_test.go 5（30 個 sub-cases） Upgrader 內部 helper 的純函式 unit 測試：splitClientProtocols（9 cases 含 bare otel-ws、prefixed、mixed）、selectFirst（7 cases 含 accept-any 語義）、cloneHeader（3 cases 含 nil-safe + 獨立性）、appProtocolFromRaw（5 cases 處理前綴剝除）、isOTelWireProtocol（6 cases 含大小寫敏感）。 Internal-package 測試（無 _test 套件後綴）— 純函式，無 I/O，無 goroutine。 conn_test.go 的高層 Scenario 測試已間接覆蓋，但每個純函式一個 9 case 表寫起來更便宜、跑得更快、bug 直接定位到具體函式而非整個 handshake 路徑。
flag_matrix_test.go 4 (1) TestWSFlagMatrix_AllCombinations — 6 個 env 組合 × 端到端 client/server 配 envelope-or-not 斷言；(2) TestWriteMessageDisabled_ByteIdenticalWithNative — 同一個 payload 並列跑過 otelgorillaws.Dial（disabled 模式）與原生 websocket.Dialer，斷言 server 收到的 byte 完全相同；(3) TestWriteMessageEnabled_NonNegotiatedStillByteIdentical — 即使 env on，若 subprotocol negotiation 失敗則 envelope 必須被抑制；(4) TestSubprotocolStripsOTelPrefix — Conn.Subprotocol() 剝除 otel-ws+ 前綴。 Internal-package test 才能直接 reset 未匯出的 wsGate。httptest server 同時用 gorilla 與 otelgorillaws upgrader。Cleanup 順序精心安排（LIFO 下 gate reset 最後跑）讓 hijacked WebSocket handler goroutine 不與 gate reset 競賽。 Byte-identical 斷言是 disabled-mode invariant 最強的可能證明 — 兩條路徑同時跑然後比對 wire bytes。若 disabled 路徑哪天意外開始包 payload（即使非預期），這個測試立即報錯。

整合測試（tests/integration/）

檔案測試數測什麼怎麼測為何測
handshake_test.go 3（含 8 sub-cases） (1) TestIntegration_Handshake_OtelWSOnlyNoAppProtocol — Scenario A；(2) TestIntegration_Handshake_SubprotocolNegotiation — Scenario B 含完整訊息 round-trip；(3) TestHandshakeScenarios_All — table-driven 全部 8 Scenarios A–H 並交叉參照 conn_test.go 中的 sibling unit 測試。 獨立 Go 模組在 tests/integration/ 下。用真實 httptest.NewServer + 各種 raw/wrapped client+server 組合。 Matrix 檔把 8 個散落的 unit 測試合進一個正式參考。任何人在程式碼任何地方看到「Scenario G」都能找到精確預期行為。
message_propagation_test.go 1 透過 envelope 的 trace context round-trip：producer span context 在 consumer 端萃取後出現。 端到端兩端都 wrap，單一訊息穿過。 Envelope 是整個 propagation 通道 — 這裡破了整個 wrapper 就失去意義。
error_fallback_test.go 1 非 envelope payload + close message 不會 crash wrapper — TryUnmarshalWire 優雅回傳 false，ReadMessage 把 raw bytes 透傳。 Server 送非 JSON payload 然後關閉；斷言 client 不錯不 panic。 真實 WebSocket 流量包含 ping/pong/close frame，客戶端也可能送未包裝 payload — 這裡的健壯性是「會崩 lib」與「能用 lib」之間的差別。
