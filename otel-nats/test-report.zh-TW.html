套件 otelnats/（core NATS）

檔案 測試數 測什麼 怎麼測 為何測
env_flags_test.go 2 兩個 gate 的真值表：(1) 兩層 tracing（OTEL_INSTRUMENTATION_GO_TRACING_ENABLED × OTEL_NATS_TRACING_ENABLED）與 (2) 三層 propagation（× OTEL_NATS_PROPAGATION_ENABLED）。涵蓋 truthy/falsy 同義詞（true/1/on/yes/""/任意；false/0/no/off/含空白）。 每列 t.Setenv + os.Unsetenv；每列後呼叫 resetNATSGateForTest() 重新啟動 sync.Once cache。 traced impl 的每個 per-call 決策最終都查這兩個 gate。真值表回歸則整個 disabled-mode invariant 靜默失效。
propagation_test.go 2 HeaderCarrier.Get/Set/Keys 對 nil nats.Header 與已填值兩種情況。 直接 struct 建構；不需 NATS server。 HeaderCarrier 是 OTel TextMapPropagator 與 NATS message header 的轉接器 — 這裡回歸則所有 Publish 注入與 Subscribe 萃取都壞。
conn_test.go 27 套件主體：W3C round-trip、Publish/PublishMsg/Request producer spans、Subscribe/QueueSubscribe consumer spans、三層 sampling gate（upstream sampled / local sampled / endpoint 設定）下的 deliver span emission、tracing-off 同時跳過 spans 與 headers、propagation-off 保留 spans 但跳過 headers、deliver-TP service name 解析、reply-link 跟隨 sampler。 透過 natssrv.NewServer 在 127.0.0.1:-1（隨機 port）啟動真 NATS server。記憶體內的 tracetest.SpanRecorder。Helper waitSpanByNameAndKind 用 require.Eventually 輪詢，容忍 async End() 競賽。 NATS 是 TCP 上的線路協定 — 只有端到端 server 測試能證明 (a) headers 確實出現在 wire bytes 與 (b) subscribe handler 確實帶著萃取出的 context 觸發。這是整個模組的負重檔。
flag_matrix_test.go 1 (8 sub-rows) 所有 8 個 (global × tracing × propagation) 組合 × 真 NATS server 端到端。每列：斷言 Conn.TracingEnabled() + PropagationEnabled() 解析為預期 gate 值；驗證 subscribe handler 在 propagation on 時才收到 traceparent header；驗證 caller TP 在 tracing on 時至少一個 wrapper span（否則零）。 每列獨立 natssrv 實例，table-driven；每列後 otelnats.ResetGatesForTest()。Span quiet-window 斷言改用 assert.Never(300ms, 10ms tick) 取代固定 sleep — 首個觀察到 span 即 fail-fast。 這是把 env 真值表與線路可觀察行為對接起來的唯一橋梁。沒有它，gate 函式可能正確但 Publish 仍注入 headers（v0.4.x tracedJSImpl.PublishMsg 無視 propagation gate 的回歸正是這類 bug）。
connect_test.go 7 (1) Connect 對無法連線 URL 回傳錯誤；(2) nil nats.Option 被過濾；(3) Close 在無 deliver-TP 時是 noop；(4) Close 遵守 3 秒 deliver-TP shutdown 上限；(5) Drain 不阻塞於慢 exporter；(6) WithTraceDestination 在 tracing-on 時保留且 (7) 在 disabled mode 被丟棄。 真 natssrv + 用 closed port（127.0.0.1:1）作 OTLP endpoint 模擬慢 exporter，無需架設 mock OTLP receiver。 連線生命週期 bug 浮現得很慢 — 失敗 shutdown 的 deliver-TP 每個 *Conn 洩漏一個 exporter batch goroutine。Bounded-time 斷言在 CI 5 秒內就抓到，而不是 production OOM 才發現。
traceevent_test.go 4 SubscribeTraceEvents（NATS 2.11+ infrastructure events）：(1) 為合成 TraceEvent payload emit 2 個 hop spans（ingress + egress）；(2) disabled mode 不發 span；(3) 損壞的 JSON payload 不會 crash 也不 emit；(4) WithTraceDestination option 透過 conn.TraceDest() round-trip。 合成 TraceEvent JSON payload 直接透過 conn.NatsConn().Publish 發送（測試夾具繞過 wrapper）。Quiet-window 負向斷言用 assert.Never(300ms)。 Infrastructure trace events 是 opt-in 功能且需要 NATS 2.11+；無此測試則唯一驅動方式是對真實 cluster 做完整 e2e。直接 payload 注入是 unit 測試 hop-emission 邏輯的唯一方法。

consumer_test.go 18 所有 consumer 模式：Fetch / FetchBytes / FetchNoWait / Consume / MessagesNext / OrderedConsumer；Consume + Fetch 的 deliver-span emission；consumer Info/CachedInfo。關鍵：TestNoGoroutineLeakAfterEarlyReturn 守 v0.3.0 加入的 MessageBatch.Stop() 契約；TestMessageBatchStopIdempotent + StopReleasesRawBatch 鎖住 cleanup 語義。TestJetStreamConsumerManagerConsumerKeepsTraceWrapper 守 Consumer-Manager 重新取得的 consumer 仍保留 trace wrapper。Propagation off / tracing off 矩陣鏡像 core-NATS 測試。 真 natssrv 開啟 JetStream（JetStream: true option）。每個測試建立自己的 stream + consumer。 JetStream 是最難的 async case — 長時間運作的 consumer 有內部狀態。Early return 洩漏的 goroutine 除非明確計數（runtime.NumGoroutine）否則在 unit test 不可見。無此守護則 v0.3.0 的破壞性變更會悄悄再次回歸。
stream_test.go 8 Stream.Info、CachedInfo、ConsumerNames、CreateConsumer、DeleteConsumer、JetStream.DeleteStream、ConsumerManager parity methods（同時存在於 Stream 與 JetStream 的平行方法）。 真 natssrv + JetStream；每測試建立 stream。 Stream 生命週期方法是 CRUD passthrough — 但每個方法都得穿過 traced wrapper，所以每個都是一行的 span-emission gate，會回歸。
